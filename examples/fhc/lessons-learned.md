# Lessons Learned — Financial Health Companion

> El agente actualiza este archivo cuando resuelve un error relevante.
> El objetivo es no repetir el mismo error dos veces.
> Los patrones recurrentes se convierten en skills en `docs/06-learning/`.
>
> Formato de cada entrada:
> Contexto → Causa raíz → Solución → Cómo detectarlo antes

---

## Formato de Entrada

```markdown
### [LL-XXXX] Título descriptivo del problema

**Fecha:** YYYY-MM-DD  
**Fase AIDLC:** Fase 1 / 2 / 3 / 4 / 5  
**Severidad:** Alta / Media / Baja  
**¿Se convirtió en skill?:** Sí — [nombre del skill] / No

**Contexto:**
Qué se estaba construyendo, qué pasó.

**Causa raíz:**
Por qué pasó — la causa real, no el síntoma.

**Solución aplicada:**
Qué se hizo para resolverlo. Pasos concretos.

**Cómo detectarlo antes:**
Señales tempranas que indican que este problema está a punto de ocurrir.

**Prevención:**
Qué cambio de proceso o checklist previene este error en el futuro.
```

---

## Registro de Lecciones

> Este archivo se puebla con el uso del proyecto.
> La primera entrada se agrega cuando ocurra el primer error documentable.

---

## Patrones que el Agente Consulta Antes de Proponer Soluciones

Aunque el registro esté vacío, el agente siempre pregunta:

1. ¿Esto se parece a algún problema que ya resolvimos?
2. ¿Hay algo en `docs/06-learning/error-patterns.md` que aplique?
3. ¿La solución que propongo introduce alguno de los riesgos que ya conocemos?

Si la respuesta a cualquiera de las tres es "sí", el agente lo menciona
explícitamente antes de proponer la solución.
